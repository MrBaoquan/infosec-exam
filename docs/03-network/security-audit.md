# 网络安全审计技术

## 1. 为什么需要审计？

想象一个场景：公司数据库被拖库，50万条用户数据泄露。事后追查——

- **没有审计**：完全不知道谁干的、什么时候、从哪里进来、拿走了什么，无法追责也无法补救
- **有审计**：日志显示"2024-03-15 02:30，IP 1.2.3.4 通过 SQL注入 获取 DBA 权限，导出 users 表"

审计的核心价值：**事后可追溯、事中可发现、事前可预警**。《网络安全法》第21条明确要求网络日志留存**不少于6个月**，这是法律强制义务。

## 2. 审计是什么？

> 🎯 **形象比喻**：审计 = 银行的"监控录像 + 交易流水 + 保安巡查记录"三位一体。

**安全审计**是对网络信息系统中**安全相关活动**进行**记录→收集→分析→审查**的完整过程，目的有四：

| 目的 | 说明 |
|------|------|
| 合规检查 | 满足等保/法规要求（日志≥6个月） |
| 事件追溯 | 出事后能还原"谁在何时做了什么" |
| 入侵检测 | 实时分析日志发现异常 |
| 取证 | 司法证据（需保证完整性与不可篡改） |

## 3. 审计系统怎么工作？（原理详解）

审计不是"记个日志"那么简单，它是一个**数据流转 pipeline**：

```
① 采集 → ② 传输 → ③ 存储 → ④ 分析 → ⑤ 报告/告警
```

### ① 数据采集（从哪来？）

审计数据有**5个来源**，对应5类审计系统：

| 审计类型 | 采集对象 | 怎么采 | 典型工具 |
|---------|---------|--------|----------|
| **网络通信审计** | 网络流量 | 交换机镜像端口/网卡混杂模式 | Tcpdump、Wireshark |
| **操作系统审计** | 系统事件 | OS内置日志机制 | Windows Event Log、Linux syslog |
| **数据库审计** | SQL操作 | 触发器/网络旁路抓包 | 数据库审计系统 |
| **应用系统审计** | 应用行为 | 应用写日志文件 | Web访问日志(Nginx/Apache) |
| **运维安全审计** | 运维命令 | 堡垒机代理所有运维操作 | 堡垒机/跳板机 |

#### 网络流量采集原理（详解）

为什么不能直接在服务器网卡上抓包？因为服务器网卡默认只处理"发给自己的包"。要抓所有流量有两种方法：

**方法一：交换机端口镜像（SPAN）**
```
                    ┌─ 服务器A ────┐
互联网 ─── 交换机 ──┤─ 服务器B ────┤
                    └─ 镜像口 ──→ 审计服务器
```
- 在交换机上配置：把某些端口的流量**复制一份**到镜像口
- 审计服务器接镜像口，收到所有流量副本
- **不影响生产**（只是复制），是生产环境首选

**方法二：网卡混杂模式**
- 网卡正常模式：只收"发给自己的帧"
- 混杂模式(Promiscuous)：收所有经过的帧
- 需要接 Hub 或交换机镜像口，否则只能收广播/发给自己的
- 用于测试环境或主机本地抓包

#### 系统日志采集：Syslog 协议详解

**Syslog** 是事实标准的日志协议，理解它要抓住3点：

**1. 传输：默认 UDP 514（明文）**
- UDP 快但不可靠（可能丢日志）
- 也可用 TCP 514 或 TLS 6514（加密，更安全）
- 生产环境敏感日志建议 TCP+TLS

**2. 格式：facility + severity + 内容**

```
<facility*8+severity> 时间 主机 标签: 内容
```

示例：
```
<34>Oct 11 22:14:15 mymachine su: 'su root' failed for user on /dev/pts/1
```

- `<34>` = facility 4(auth) × 8 + severity 2(critical) = 34
- **facility**（来源）：0(kernel) ~ 7(news)，如 auth(4)、mail(2)、cron(9)
- **severity**（严重度）：0(emerg最严重) ~ 7(debug)，常考的有：
  - 0 emerg：系统不可用
  - 1 alert：必须立即处理
  - 2 crit：严重情况
  - 3 err：错误
  - 4 warning：警告
  - 6 info：信息
  - 7 debug：调试

> 💡 **考试技巧**：severity 0 最严重、7 最轻微，数字越小越严重，别记反。

**3. 集中收集：rsyslog / 日志服务器**
- 每台设备配置 rsyslog 转发到中心日志服务器
- 中心服务器统一存储、分析、告警
- 工具：rsyslog、syslog-ng、ELK(Elasticsearch+Logstash+Kibana)

### ② 数据传输与存储

- 传输：Syslog(UDP/TCP) 或专用协议
- 存储：集中存储到日志服务器/SIEM系统
- **关键：防篡改**——攻击者第一件事就是删日志
  - 分权管理：审计员与系统管理员**职责分离**，审计员不能改系统，管理员不能删日志
  - 只追加(Append-only)：日志只许追加不许删改
  - 加密存储：敏感日志加密
  - 哈希校验：定期对日志计算哈希，事后对比发现是否被篡改
  - 异地备份：日志备份到另一台服务器

### ③ 数据分析（怎么发现问题？）

| 方法 | 说明 | 示例 |
|------|------|------|
| **字符串匹配** | 关键字检索 | grep "failed password" 找爆破 |
| **全文搜索** | 大量日志快速检索 | ELK 的 Kibana 搜索 |
| **数据关联** | 多源日志关联 | 同一IP短时间内多次登录失败+成功 |
| **统计报表** | 趋势分析 | 某账号凌晨3点大量下载，异常 |
| **可视化** | 图表展示 | 登录热力图、流量趋势 |

**关联分析是核心**——单条日志看不出问题，多条关联才能还原攻击：
```
日志1: 02:00  IP 1.2.3.4 登录失败 × 1000次  ← 爆破
日志2: 02:05  IP 1.2.3.4 登录成功            ← 破解成功
日志3: 02:10  IP 1.2.3.4 导出 users 表       ← 拖库
→ 关联后：完整攻击链
```

## 4. 实操：Tcpdump 抓包实战

Tcpdump 是最经典的命令行抓包工具，考试常考命令参数。

### 基本语法

```bash
tcpdump -i <接口> [过滤表达式] [-w 文件] [-r 文件] [-c 数量] [-n]
```

### 常用参数

| 参数 | 说明 | 示例 |
|------|------|------|
| `-i` | 指定网卡 | `-i eth0` |
| `-w` | 保存到文件 | `-w capture.pcap` |
| `-r` | 读取文件 | `-r capture.pcap` |
| `-c` | 抓N个包后停止 | `-c 100` |
| `-n` | 不解析域名(快) | `-n` |
| `-nn` | 不解析端口名 | `-nn` |

### 过滤表达式（BPF语法，高频考点）

```bash
# 按主机
tcpdump host 192.168.1.1                # 抓与1.1通信的所有包
tcpdump src 192.168.1.1                 # 源是1.1
tcpdump dst 192.168.1.1                 # 目的是1.1

# 按端口
tcpdump port 80                         # 80端口
tcpdump port 53                         # DNS

# 按协议
tcpdump tcp                             # TCP包
tcpdump icmp                            # ICMP包(ping)

# 组合(与/或/非)
tcpdump 'tcp port 80 and host 192.168.1.1'    # 80端口且与1.1通信
tcpdump 'tcp port 80 or tcp port 443'          # 80或443
tcpdump 'not port 22'                          # 排除SSH
```

### 实战场景

**场景1：排查某IP的网络活动**
```bash
tcpdump -i eth0 host 1.2.3.4 -nn
```

**场景2：保存流量事后分析**
```bash
tcpdump -i eth0 -w attack.pcap
# 事后用 Wireshark 打开 attack.pcap 图形化分析
```

**场景3：只抓前100个TCP包**
```bash
tcpdump -i eth0 tcp -c 100
```

> 💡 **Tcpdump vs Wireshark**：Tcpdump 命令行抓包(轻量，服务器用)；Wireshark 图形化分析(功能强，桌面用)。两者都用 pcap 格式，可互通。

## 5. 易错辨析（考试怎么考）

### 易错1：审计 vs 日志
- **日志**：原始记录（一条条事件）
- **审计**：对日志的**收集+分析+审查**的完整过程
- 日志是审计的**输入**，审计是日志的**升华**

### 易错2：审计员与管理员的职责
- 系统管理员：管系统，**不能删改日志**
- 审计管理员：管日志，**不能改系统配置**
- **必须职责分离**，否则管理员作恶后删日志就无据可查

### 易错3：日志留存期限
- 《网络安全法》：**≥6个月**
- 注意是"6个月"不是"3个月"或"1年"

### 易错4：TCP vs UDP syslog
- 默认 UDP 514（快但可能丢）
- TCP 514 / TLS 6514（可靠/加密，敏感场景必选）
- 考题问"如何保证日志不丢/防窃听"→选 TCP+TLS

## 6. 真题示例及解析

### 真题1（上午·单选）

> 根据《网络安全法》，网络日志的留存期限不少于？
> A. 3个月  B. 6个月  C. 1年  D. 2年

**答案：B**

**解析**：《网络安全法》第21条明确规定网络运营者留存网络日志不少于6个月。这是高频记忆点，常以选择题出现。易错选A(3个月，混淆了部分行业规范)或C(1年)。

---

### 真题2（上午·单选）

> Syslog 协议默认使用的传输协议和端口号是？
> A. TCP 514  B. UDP 514  C. TCP 161  D. UDP 161

**答案：B**

**解析**：Syslog 默认走 **UDP 514**（明文、快但不可靠）。TCP 514/TLS 6514 是增强版（可靠/加密）。注意区分：161 是 SNMP 的端口，别混淆。考"如何保证日志不丢/防窃听"时选 TCP+TLS。

---

### 真题3（上午·单选）

> 在 Linux 系统中，Syslog 日志消息的严重等级（severity）数值与严重程度的关系是？
> A. 数值越大越严重  B. 数值越小越严重  C. 偶数严重  D. 与数值无关

**答案：B**

**解析**：severity 范围 0-7，**0 最严重（emerg），7 最轻微（debug）**，数值越小越严重。易记反。例如 emerg=0、alert=1、crit=2、err=3、warning=4、info=6、debug=7。

---

### 真题4（上午·单选）

> 下列关于交换机端口镜像（SPAN）的描述，正确的是？
> A. 镜像口会修改原始流量  B. 镜像不影响生产流量，只是复制一份  C. 镜像口只能镜像一个端口  D. 镜像口必须接 Hub

**答案：B**

**解析**：SPAN 是把指定端口流量**复制一份**到镜像口，审计设备接镜像口即可收到流量副本。**不影响生产**（只读复制），是生产环境流量采集首选。可镜像多个源端口到一个镜像口。

### 真题5（下午·简答）

> 某公司发生数据泄露，安全员在排查时发现日志已被攻击者清除。请从审计安全角度，给出至少3条防止日志被篡改的措施。

**参考答案**（分点作答，每点1分）：

① **职责分离**：审计管理员与系统管理员权限分离，管理员无法删除日志，审计员无法修改系统配置（2分）

② **只追加存储**：日志文件设置为只追加(append-only)模式，禁止修改和删除已写入的日志记录（2分）

③ **异地备份**：日志实时同步到独立的日志服务器或异地存储，即使本机日志被删，异地仍有副本（2分）

④ **完整性校验**：定期对日志文件计算哈希值并保存，事后对比哈希可发现是否被篡改（2分）

⑤ **加密存储**：敏感日志加密存储，即使被窃取也无法直接读取（1分）

**评分要点**：每条措施需写出"做法+目的"，只写"防篡改"不给分，需具体到"怎么做"。

### 真题6（下午·简答）

> 请说明 Tcpdump 工具中以下命令的作用，并给出一个用于捕获指定 IP 到指定端口的流量的完整命令。

(1) `-i` 参数的作用
(2) `-w` 参数的作用
(3) 写出捕获来自 192.168.1.100 且目标端口为 80 的 TCP 流量的命令

**参考答案**：

(1) `-i` 指定抓包的网络接口（网卡），如 `-i eth0` 表示在 eth0 接口抓包（2分）

(2) `-w` 将抓到的数据包保存到指定文件（pcap格式），如 `-w capture.pcap`，便于事后用 Wireshark 分析（2分）

(3) 完整命令：
```bash
tcpdump -i eth0 'src 192.168.1.100 and tcp dst port 80' -w web.pcap
```
（3分，需包含接口、过滤表达式、可选保存）

**评分要点**：过滤表达式需正确使用 `src`、`and`、`tcp dst port` 关键字组合。

## 7. 考点速记
- 审计5类：网络/操作系统/数据库/应用/运维
- 网络流量采集：**端口镜像(SPAN)** 生产首选
- Syslog：**UDP 514**，facility×8+severity，severity **0最严重7最轻**
- 采集→传输→存储→分析→告警 五步 pipeline
- 审计数据保护：**分权管理、只追加、加密、哈希校验、异地备份**
- 日志留存 **≥6个月**（网安法）
- 审计员与管理员**职责分离**
- Tcpdump：`-i`指定网卡、`-w`保存、`-r`读取、`-c`数量、`-n`不解析
- 关联分析：多源日志关联还原攻击链
- Tcpdump 命令行抓包；Wireshark 图形化分析；pcap 格式互通
