# 信息安全工程

> 🎯 **形象比喻**：安全工程像"盖安全大楼"——不能盖完再装防盗门（事后补），要从设计图阶段就考虑安全（**安全左移**）。

## 系统安全工程

信息安全工程不是在上线前做一次扫描，而是把安全活动贯穿系统**全生命周期**：

```
需求 → 设计 → 实现 → 测试 → 部署 → 运维 → 退役
```

> **安全左移**是尽早发现和处理问题。它不表示运维阶段不再负责安全，而是让安全需求、设计和验证更早开始，并在后续阶段持续闭环。

### 为什么需要生命周期方法

某医院上线互联网预约系统。如果团队只在上线前安装防火墙，仍可能遗漏三类问题：

- 需求阶段没有定义病历数据的访问和留存要求；
- 设计阶段没有识别越权访问和接口滥用威胁；
- 退役阶段直接丢弃旧磁盘，导致敏感数据泄露。

安全工程要回答的不只是“买什么设备”，而是“每个阶段由谁完成什么活动、留下什么证据、如何验证风险已被处理”。

### 各阶段活动与产物

| 阶段 | 关键活动 | 典型产物 |
|------|----------|----------|
| **需求** | 识别资产、合规义务、业务影响和安全目标；定义可验证的安全需求 | 资产清单、数据分类分级、安全需求规格说明书 |
| **设计** | 划分信任边界，进行威胁建模，选择认证、授权、加密、审计和容灾方案 | 安全架构图、数据流图、威胁模型、安全设计说明书 |
| **实现** | 安全编码、依赖管理、密钥管理、代码审查和配置基线 | 代码审查记录、依赖清单、配置基线 |
| **测试** | 验证安全需求，执行漏洞扫描、代码审计、渗透测试和修复复测 | 测试用例、漏洞报告、整改与复测记录 |
| **部署** | 最小化安装、关闭不必要服务、配置权限、证书、日志、备份和监控 | 部署检查表、基线核查记录、上线审批记录 |
| **运维** | 监控告警、漏洞与补丁管理、账号复核、事件响应、备份恢复演练 | 审计日志、补丁台账、应急记录、演练报告 |
| **退役** | 撤销账号与证书、迁移和归档数据、清除介质、更新资产清单 | 退役审批、数据迁移记录、介质清除或销毁证明 |

### 怎么把安全需求写成可验证要求

“系统必须安全”无法测试。合格的安全需求需要明确对象、条件和结果：

| 模糊要求 | 可验证要求 |
|----------|------------|
| 保护用户数据 | 用户敏感数据传输使用 TLS，存储使用经批准的加密方案 |
| 防止越权 | 普通用户只能读取自己的预约记录，管理员操作必须记录审计日志 |
| 系统要可靠 | 核心预约服务达到约定可用性，备份恢复时间满足业务影响分析结果 |

> 💡 考题要求“提出安全需求”时，应写成可检查、可测试的控制要求，不要只写“加强安全”“提高意识”。

### 贯穿案例：医院预约系统

1. **需求**：病历和身份证号属于敏感数据；患者只能查看本人记录；关键操作需要留痕。
2. **设计**：划分互联网区、应用区和数据区；使用统一认证、最小权限和数据库审计；对接口做限速。
3. **实现**：参数化查询防 SQL 注入；服务端校验资源归属防越权；密钥不写入源代码。
4. **测试**：覆盖身份冒用、越权、注入、敏感信息泄露和恢复演练；修复后必须复测。
5. **部署**：关闭调试模式和默认账号；配置证书、日志、监控与备份告警。
6. **运维**：定期复核权限和依赖漏洞；发生事件时按准备、检测、抑制、根除、恢复、总结闭环。
7. **退役**：撤销系统证书和服务账号；完成数据迁移；对旧介质安全擦除或物理销毁。

## SSE-CMM 安全工程能力成熟度模型

描述组织安全工程过程能力，**5 个等级**：

| 级别 | 名称 | 特征 |
|------|------|------|
| 1 | **非正式** | 依赖个人英雄主义，不可重复 |
| 2 | **计划与跟踪** | 可重复，有基本计划 |
| 3 | **充分定义** | 标准化流程，组织级 |
| 4 | **量化控制** | 可度量，数据驱动 |
| 5 | **持续改进** | 优化改进，自适应 |

> 与 CMM 软件能力成熟度模型对应，SSE-CMM 专注安全工程。

## CC 通用准则（ISO/IEC 15408）

- 国际信息安全产品评估标准
- 三部分：
  - 第1部分：简介和一般模型
  - 第2部分：**安全功能要求**（SFR）
  - 第3部分：**安全保证要求**（SAR）

### EAL 评估保证级

| 级别 | 保证程度 |
|------|----------|
| EAL1 | 功能测试（最低） |
| EAL2 | 结构测试 |
| EAL3 | 方法测试与检查 |
| EAL4 | 系统设计测试与复查（**最高商业实用**） |
| EAL5 | 半形式化设计与测试 |
| EAL6 | 半形式化验证设计与测试 |
| EAL7 | 形式化验证（最高） |

> EAL4 是商业产品中通常较常见的高保证级别；EAL5-7的应用较少，具体取决于产品、保护轮廓和评估目标，不能简单等同于某一行业。

## 安全设计原则（Saltzer-Schroeder）

| 原则 | 含义 |
|------|------|
| **经济机制** | 简单可理解 |
| **失败安全** | 失败时进入安全状态（默认拒绝） |
| **默认安全** | 默认配置安全 |
| **纵深防御** | 多层防护 |
| **最小权限** | 仅授予必要权限 |
| **职责分离** | 关键操作多人 |
| **完全调解** | 每次访问都检查 |
| **开放设计** | 安全不依赖保密（Kerckhoffs 原则） |
| **心理可接受** | 安全措施易用 |
| **最薄弱环节** | 重视最弱点 |

## 可信计算

| 概念 | 说明 |
|------|------|
| **TPM** | 可信平台模块（硬件根可信） |
| **TCM** | 国产可信密码模块 |
| **信任链** | CRTM→BIOS→OS→应用，逐级度量 |
| **远程证明** | 向第三方证明平台可信状态 |

> 信任根：CRTM（核心度量根），硬件固化不可篡改。

## 5. 易错辨析

### 易错1：SSE-CMM 5级
- 1非正式→2计划跟踪→3充分定义→4量化控制→5持续改进

### 易错2：CC/EAL
- EAL1~EAL7
- **EAL4**：商业产品中通常较常见的高保证级别
- EAL5-7：军方/高安全

### 易错3：Kerckhoffs原则
- 安全不依赖算法保密（开放设计）
- 安全只依赖密钥

### 易错4：安全设计原则
- 默认安全/纵深防御/失败安全/最小权限/职责分离/开放设计

### 易错5：安全左移
- 左移是把需求分析、威胁建模和安全测试提前，不是只在开发阶段做安全
- 上线后仍需监控、补丁、应急和恢复演练
- 退役阶段仍需处理账号、证书、数据和介质

## 6. 真题示例及解析

### 真题1（上午·单选）

> SSE-CMM 共有几个等级？
> A. 3级  B. 4级  C. 5级  D. 7级

**答案：C**

**解析**：SSE-CMM共5级：1非正式→2计划跟踪→3充分定义→4量化控制→5持续改进。

---

### 真题2（上午·单选）

> CC 通用准则中，商业产品通常达到的最高级别是？
> A. EAL1  B. EAL3  C. EAL4  D. EAL7

**答案：C**

**解析**：EAL4在商业产品中较常见，EAL5-7要求更高、应用相对少见；具体等级应结合保护轮廓和评估目标判断。

---

### 真题3（上午·单选）

> Kerckhoffs 原则的核心思想是？
> A. 算法应保密  B. 安全不应依赖算法保密  C. 密钥应公开  D. 加密越复杂越安全

**答案：B**

**解析**：安全应基于密钥保密而非算法保密。假设攻击者知道算法，安全只依赖密钥。

---

### 真题4（上午·单选）

> "系统异常时进入安全状态，默认拒绝访问"属于？
> A. 最小权限  B. 纵深防御  C. 失败安全  D. 默认安全

**答案：C**

**解析**：失败安全(Fail Safe)：异常时进入安全状态，默认拒绝。

---

### 真题5（下午·简答）

> 某单位开发网上业务系统，计划在编码完成后统一进行漏洞扫描。请指出该做法的问题，并按系统生命周期列出四项应提前或持续开展的安全活动。

**参考答案与评分点：**

1. 只在编码完成后扫描属于事后补救，安全介入过晚，难以及时发现需求和架构层问题。（2分）
2. 需求阶段识别资产、合规要求和安全目标，形成可验证的安全需求。（2分）
3. 设计阶段开展威胁建模、信任边界分析和安全架构设计。（2分）
4. 实现阶段执行安全编码、依赖管理和代码审查。（2分）
5. 测试、部署和运维阶段持续开展安全测试、基线核查、监控、补丁与应急响应。（答出两项得2分）

**答题方法**：先指出“介入太晚”，再按阶段写“活动+产物或目标”。只罗列防火墙、WAF、杀毒软件，不能完整回答生命周期问题。

## 7. 考点速记

- SSE-CMM **5级**：1非正式→2计划跟踪→3充分定义→4量化控制→5持续改进
- CC 三部分：简介/功能要求/保证要求；**EAL1~EAL7**
- **EAL4** 商业最高实用级；EAL5-7 军方级
- 安全原则：**默认安全、纵深防御、失败安全、最小权限、职责分离、开放设计**
- Kerckhoffs：**安全不依赖算法保密**（应假设攻击者知道算法）
- TPM 硬件根可信；信任链逐级度量
- 生命周期：需求→设计→实现→测试→部署→运维→退役，每阶段都有安全活动和交付物
- 安全需求必须可验证；安全左移不等于上线后停止安全工作
